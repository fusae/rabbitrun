用了两个月 OpenClaw，我终于受不了了。不是它不好——它确实帮我从零搭起了 AI 辅助工作流。但用着用着，几个问题越来越明显：回复经常卡顿 30 秒以上，多轮对话上下文说着说着就丢了，最烦的是每次换项目都要重新配置环境。我是一个人运营，不需要什么多 Agent 协作、任务编排这些花活。我要的就是：打开飞书，跟 AI 说一句话，它在正确的项目目录里干活，回复快一点，别丢上下文。

## 一次偶然的重构

上周末我在折腾 opencode 的本地服务模式，发现它原生就支持 opencode serve——起一个 HTTP 服务，任何客户端都能调用。我当时就想：能不能让飞书直接连它？答案是能，而且比我预想的简单得多。整个方案就三层：飞书负责收发消息，本地跑一个 opencode 服务，中间加一个 bridge 程序把两边接起来。bridge 做的事情很纯粹——收到飞书消息，转发给 opencode，拿到回复再发回飞书。没有中间件，没有消息队列，没有容器。一个 Node.js 进程，常驻后台，完事。

## 实际体验差多少

换完之后第一天，最直观的感受是**快**。之前 OpenClaw 从我发消息到收到回复，平均要 20-40 秒，有时候更长。现在基本 5-10 秒就开始回，复杂任务也不会超过30 秒。因为 opencode 就跑在我自己电脑上，没有中间转发链路的开销。第二个感受是**省心**。我所有项目都在 ~/Projects 目录下，bridge 会自动扫描，第一次聊天时弹一个项目列表，点一下就绑定了。后续所有消息都在那个项目目录里执行。换项目？发个 /switch 就行。第三个感受是**省钱**。之前 OpenClaw 每次对话都要经过中转、规划和上下文包装，来回好几轮 agent 才落地。现在飞书消息直达 opencode 会话，少了中间这层转发和重复的系统提示，token 消耗明显降下来了。同样的事，花更少的 token 就干完了。之前用 OpenClaw，光配置环境就折腾了我两个下午。这次从零搭建，包括去飞书开放平台建应用，总共不到 40 分钟。

## 最关键的一步

如果你也想试，核心就一件事：去飞书开放平台创建一个自建应用，设好权限。开启 Bot 能力，订阅消息接收事件，接入方式选长连接。然后把 App ID 和 App Secret 填到 bridge 的配置文件里，npm run dev 一跑，完事。不需要服务器，不需要公网 IP，不需要域名。飞书的长连接是从内网主动连出去的，你的电脑在哪，AI 就跟到哪。

## 谁该换，谁不用换

说实话，如果你是团队协作、需要多 Agent 并行、依赖 OpenClaw 的插件生态，没必要换。但如果你跟我一样——一个人，用 AI 辅助内容创作和项目管理，日常就是飞书 + 代码——那直连方案明显更轻、更快、更可控。OpenClaw 像是一个全功能厨房，什么灶都有。但我只需要一个电磁炉。
