 一开始，我有一个 OpenClaw 实例，从选题到写稿到发布，全流程都能跑通。能用，但我总觉得不够"专业"。我看到各种 AI 自动化的架构图，都是多个 Agent 分工协作。我想，既然要做内容自动化，是不是也该搞个"内容工厂"？于是我开始折腾多 Agent。

---

## 第一次尝试：部署多个独立的 OpenClaw 实例

我的想法很简单：• 一个 bot 专门做选题• 一个 bot 专门写稿• 一个 bot 专门配图• 一个 bot 专门发布• 再来一个 bot 做数据复盘听起来很合理对吧？分工明确，各司其职。我花了时间把这几个 OpenClaw 实例都部署好了，每个都绑定到飞书的不同 bot。然后，问题来了。**它们之间怎么协作？**我在飞书群里让 A bot @ B bot，想让选题 bot 把选题传给写稿 bot。等了 5 分钟，B bot 一点反应都没有。我以为是配置问题，又改了一版，还是没反应。我开始怀疑是权限问题，翻了半天飞书文档，最后才发现：**不是我配错了，是飞书这套机制压根就不支持 bot 之间互相 @。**那一刻我才意识到，我不是在解决技术问题，我是在跟平台限制死磕。

---

## 第二次尝试：改成多 agent 架构

既然多个独立 bot 不行，那就改成一个 OpenClaw 实例，里面跑多个 agent。我重新设计了架构：• Mac虾（主 agent）：负责调度和协调• 网虾（子 agent）：负责选题研究• 飞虾（子 agent）：负责写稿• 图虾（子 agent）：负责配图• 分虾（子 agent）：负责发布这次架构上没问题了，主 agent 可以调用子 agent。但新的问题来了：**配环境。**多 agent 的配置比我想象的复杂得多。每个 agent 要单独配置，主 agent 和子 agent 之间的通信规则要写清楚，权限要设置好。我折腾了很久，还在配环境阶段，连完整的流程都没跑通。那天我突然想起来，**我最开始那个单 OpenClaw 实例，从选题到输出，早就能跑通了。**

---

## 我停下来，问了自己一个问题

**我真的需要多 Agent 吗？**仔细想了一下我的工作流：1. 早上看热点，选 1-2 个选题2. 写一个 1000 字左右的稿子3. 配图、排版、发布4. 第二天看看数据这本质上是一个顺序执行的任务，不是并行协作的项目。我需要的不是 5 个"专家"，而是一个能帮我把一件事从头到尾做完的助手。

---

## 最后我回到了起点

现在我用的还是最开始那个方案：**1 个 OpenClaw 实例。**它知道完整的工作流，有上下文记忆，能自己决定什么时候该搜索、什么时候该写稿、什么时候该发布。不需要配置复杂的 agent 通信规则，不需要担心哪个环节掉链子，不需要花时间调试多个故障点。**效果怎么样？**能用，稳定，每天 1 篇内容照常产出。而我那套多 agent 架构，配了很久，还没跑通完整流程。

---

## 什么时候真的需要多 Agent？

后来我也没得出"多 Agent 没用"这种结论。真要同时盯很多数据源，或者真有几种完全不同的能力要一起配合，多 Agent 当然有意义。问题是我当时做的事情根本没那么复杂。硬上多 Agent，只是在给自己制造新的维护成本。

---

## 我的结论

不是多 Agent 不行，是我这点工作量，根本配不上那套架构。一个人能干完的事，硬要拆成 5 个角色，本质上是在给自己制造新的故障点。至少对我现在这个阶段来说，1 个 OpenClaw 实例已经够用了。
