怎么做到的？QClaw 每天白送 4000 万 token。对，你没看错，每天 4000 万。问题是，这么多额度只能在装了 QClaw 的那台机器上用。我本地 Mac 上装了 QClaw，一天根本用不完 4000 万 token。但我还有几只小龙虾跑在服务器上做定时任务、跑在测试机上做实验，它们都没额度。这不是浪费吗？能不能让其他小龙虾也白嫖 Mac 上这份额度？答案是：可以。而且根据你的小龙虾部署在哪里，有两种方案：

---

## 方案一：你有服务器权限（推荐）

**适用场景：**• 你的小龙虾部署在自己的服务器上• 你有 root 权限，能安装软件• 比如：阿里云、腾讯云、自己的 VPS**核心思路：把 QClaw 当成"共享 token 池"**QClaw 本地其实会起一个代理服务，地址类似：http://127.0.0.1:19000/proxy/llm
它对外表现为一个 OpenAI-compatible 的接口。问题是这个端口只监听本机 127.0.0.1，其他机器访问不了。解决方案：1. 用 Tailscale 把所有机器拉进一个私有网络2. 用 socat 把 Mac 上的 127.0.0.1:19000 转发到 Tailscale 网络里3. 其他小龙虾直接连这个地址，就能用 Mac 上的 QClaw 额度了不需要公网 IP，不需要路由器配置，不需要逆向任何接口。

### 第一步：Mac 端（有 QClaw 的那台）

安装 Tailscale 和 socat：brew install tailscale socat
启动 Tailscale：sudo /opt/homebrew/opt/tailscale/bin/tailscaled
另开一个终端，登录：tailscale up
浏览器完成登录后，查看分配的 IP：tailscale ip -4
会得到一个类似 100.109.xx.xx 的地址。然后用 socat 把 QClaw 的本地代理转发到这个 IP：TS_IP="$(tailscale ip -4 | head -n1)"
socat TCP-LISTEN:19001,fork,reuseaddr,bind=$TS_IP TCP:127.0.0.1:19000
这时候，Tailscale 网络里的其他机器就能访问：http://100.109.xx.xx:19001/proxy/llm

### 第二步：服务器端（想白嫖的小龙虾）

安装 Tailscale：curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
登录完成后，确认能 ping 通 Mac：ping 100.109.xx.xx
先用 curl 测试一下：curl -i \
 -H 'Authorization: Bearer __QCLAW_AUTH_GATEWAY_MANAGED__' \
 -H 'Content-Type: application/json' \
 -d '{"model":"modelroute","messages":[{"role":"user","content":"hi"}],"stream":false}' \
 http://100.109.xx.xx:19001/proxy/llm/chat/completions
如果返回正常的 JSON 响应，说明链路通了。然后配置 OpenClaw：openclaw onboard --non-interactive \
 --auth-choice custom-api-key \
 --custom-provider-id qclaw \
 --custom-base-url "http://100.109.xx.xx:19001/proxy/llm" \
 --custom-model-id "modelroute" \
 --custom-api-key "__QCLAW_AUTH_GATEWAY_MANAGED__" \
 --custom-compatibility openai \
 --accept-risk
设置为默认模型：openclaw models set qclaw/modelroute
检查：openclaw models status
看到默认模型是 qclaw/modelroute，就配置完成了。

### 注意事项

**1. Mac 上的 QClaw 必须开着**这套方案是让其他小龙虾通过 Mac 上的 QClaw 中转，所以 QClaw 关了，其他小龙虾就用不了。**2. 不要把端口绑到 0.0.0.0**socat 转发的时候，只绑定 Tailscale IP（bind=$TS_IP），不要绑 0.0.0.0。否则会暴露给所有网卡，不安全。**3. Tailscale IP 一般不会变**日常换 Wi-Fi 不会影响 Tailscale IP。但如果设备重建、退出 Tailnet 重新加入，可能会变。想更稳的话，可以用 Tailscale 的 MagicDNS 主机名。**4. 想要开机自启动**临时测试通过后，可以把 tailscaled 和 socat 都改成后台服务。Mac 上可以用 Homebrew Services 和 LaunchAgent，服务器上可以用 systemd。

## 方案二：你没有服务器权限（一键部署）

**适用场景：**• 你的小龙虾是一键部署的• 你没有服务器 root 权限• 你访问不了服务器，只能通过网页/对话配置**核心思路：用 Cloudflare Tunnel 把本地 QClaw 安全地暴露到公网**以飞书一键部署为例：飞书 OpenClaw (公网)
 │
 │ HTTPS + Service Token 认证
 ▼
Cloudflare Access (认证层)
 │
 ▼
Cloudflare Tunnel (加密隧道)
 │
 ▼
本地 QClaw 代理 (127.0.0.1:19000)
关键点：1. **Cloudflare Tunnel** 把本地服务暴露成公网 HTTPS 地址2. **Cloudflare Access** 加一层认证，防止被白嫖3. **Service Token** 让 API 调用能通过认证

### 前置条件

• 本地有 QClaw 在运行• 有一个域名（可以在 Cloudflare 买，最便宜有几刀的）• 飞书部署的 OpenClaw 支持自定义 API 端点配置

### 第一步：注册 Cloudflare 并购买域名

1. 访问 https://dash.cloudflare.com 注册账号2. 左侧菜单 **Domain Registration**3. 搜索并购买一个域名（比如 yourname.com）4. 域名会自动添加到你的账号，DNS 已托管

### 第二步：安装并配置 Cloudflare Tunnel

在你的 QClaw 机器上：# macOS
brew install cloudflared

# 登录 Cloudflare（会弹浏览器授权）
cloudflared tunnel login

# 创建 Tunnel
cloudflared tunnel create qclaw
会输出 Tunnel ID，类似：abc123-def456-...创建配置文件 ~/.cloudflared/config.yml：tunnel:<你的Tunnel-ID>
credentials-file:/Users/你的用户名/.cloudflared/<Tunnel-ID>.json

ingress:
-hostname:qclaw.你的域名.com
service:http://127.0.0.1:19000
-service:http_status:404
绑定 DNS：cloudflared tunnel route dns qclaw qclaw.你的域名.com
启动 Tunnel：# 前台测试
cloudflared tunnel run qclaw

# 或安装为系统服务（长期运行）
sudo cloudflared service install
现在 https://qclaw.你的域名.com 已经能访问到本地 QClaw 了，但任何人都能访问。

### 第三步：配置 Cloudflare Access 认证

#### 3.1 创建应用

1. 访问 https://one.dash.cloudflare.com2. 左侧 **Access → Applications → Add an application**3. 跳过引导页，直接找到 **Add an application** 按钮4. 填写：• Subdomain: qclaw• Domain: 你的域名.com• **Application name**: Qclaw Proxy• **Session Duration**: 24 hours• 点 **Add public hostname**：

#### 3.2 创建邮箱策略（可选，用于浏览器访问）

在同一页面，**Add a policy**：• **Policy name**: Only Me• **Action**: Allow• **Include** → **Emails** → 填你的邮箱

#### 3.3 创建 Service Token（用于 API 调用）

1. 左侧 **Access → Service Auth → Add a service token**2. **Name**: openclaw3. **Duration**: Non-expiring4. 点 **Generate token**，复制：• CF-Access-Client-Id• CF-Access-Client-Secret**重要**：这两个值只显示一次，保存好。

#### 3.4 添加 Service Token 策略

回到 **Access → Applications → Qclaw Proxy**，点 **Add a policy**：• **Policy name**: Service Token• **Action**: Service Auth• **Include** → **Service Token** → 勾选 openclaw保存。

### 第四步：配置飞书 OpenClaw

在飞书部署的 OpenClaw 机器上，执行：# 一次性配置所有参数
openclaw config set models.providers.qclaw '{
 "baseUrl": "https://qclaw.你的域名.com/proxy/llm",
 "api": "openai-completions",
 "apiKey": "__QCLAW_AUTH_GATEWAY_MANAGED__",
 "models": ["modelroute"],
 "headers": {
 "CF-Access-Client-Id": "你的Client-ID",
 "CF-Access-Client-Secret": "你的Client-Secret"
 }
}' --json

# 切换到 Qclaw 模型
openclaw models set qclaw/modelroute

# 重启 gateway
openclaw gateway restart

### 第五步：验证

测试一下能不能正常对话：curl https://qclaw.你的域名.com/proxy/llm/chat/completions \
 -H "CF-Access-Client-Id: 你的Client-ID" \
 -H "CF-Access-Client-Secret: 你的Client-Secret" \
 -H "Authorization: Bearer __QCLAW_AUTH_GATEWAY_MANAGED__" \
 -H "Content-Type: application/json" \
 -d '{"model":"modelroute","messages":[{"role":"user","content":"hi"}],"stream":false}'
如果返回正常的 JSON 响应，说明配置成功。在飞书 OpenClaw 里直接对话，应该也能正常使用了。

### 常见问题

**1. 返回 302 重定向到登录页**原因：Service Token 策略没有正确应用到应用上。解决：1. 检查 **Access → Applications → Qclaw Proxy** 的 Policies 列表2. 确保有一个 Action 为 **Service Auth** 的策略3. 该策略的 Include 规则里要包含你创建的 Service Token**2. 返回 403 Forbidden**原因：Service Token 的 Client ID 或 Secret 填错了。解决：重新检查配置文件里的 CF-Access-Client-Id 和 CF-Access-Client-Secret。**3. 连接超时**原因：• QClaw 没有运行• cloudflared tunnel 没有运行• DNS 还没生效（新域名需要等几分钟）解决：# 检查 QClaw 是否运行
ps aux | grep -i qclaw

# 检查 tunnel 是否运行
ps aux | grep cloudflared

# 重启 tunnel
cloudflared tunnel run qclaw

### 成本

• **Cloudflare Tunnel**：免费• **Cloudflare Access**：50 用户以内免费• **域名**：几刀一年• **QClaw Token**：免费 4000 万/天总成本：**仅域名费用**

## 两种方案对比

对比项方案一：Tailscale方案二：Cloudflare**适用场景**有服务器权限没有服务器权限（飞书一键部署）**配置难度**简单中等**成本**免费域名成本**安全性**私有网络，不暴露公网公网暴露，但有认证保护**稳定性**高高**速度**快（私有网络直连）稍慢（经过 Cloudflare）**推荐：**• 如果你有服务器权限，优先用方案一（Tailscale）• 如果你是飞书一键部署，只能用方案二（Cloudflare）
